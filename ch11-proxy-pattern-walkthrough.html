Oh No!
"I need to monitor them REMOTELY!"
The CEO doesn't want to walk up to each machine. He wants to sit at his desk and monitor machines over the network. Our monitor only works locally — it needs a direct reference to the GumballMachine object in the same JVM.
// The problem: our monitor is TIGHTLY COUPLED to the concrete class public class GumballMonitor { GumballMachine machine; // ← Can't get a reference to a remote object! // Objects live in DIFFERENT JVMs on different machines! public GumballMonitor(GumballMachine machine) { this.machine = machine; // ← Violates "program to interfaces" } // We're coding to a CONCRETE class public void report() { machine.getLocation(); // ← These look like local calls... machine.getCount(); // but we need them to work over the NETWORK machine.getState(); // ← Violates "encapsulate what varies" — } // the network communication isn't encapsulated! }
Let's diagnose this with the OO principles we already know:
❌ Program to interfaces, not implementations
GumballMonitor depends on the concrete GumballMachine class. We can't swap in a remote stand-in.
❌ Encapsulate what varies
The location of the object (local vs. remote) will vary, but it's not encapsulated. The monitor assumes local access.
❌ Depend on abstractions, not concrete classes
The monitor is locked to GumballMachine. We can't substitute anything that talks over a network.
Three principles violated. We need a way for the monitor to talk to a machine that lives in a DIFFERENT JVM, on a DIFFERENT physical computer — without changing the monitor's code much at all.
Design Principle
Apply: "Program to Interfaces, Not Implementations"
Instead of coding to the concrete GumballMachine, we create a remote interface that defines what operations can be called remotely.
OO Principle
"Program to interfaces, not implementations."

The Revelation
Wait... This Is the Proxy Pattern!
By applying three principles — program to interfaces, encapsulate what varies, and favor composition — we independently arrived at a well-known design pattern.
The Proxy Pattern
"Provide a surrogate or placeholder for another object to control access to it."
Think of it like a corporate bodyguard. The CEO (client) wants to talk to a celebrity (real subject). The bodyguard (proxy) has the same communication abilities but controls who gets through, handles logistics, and may even stand in when the celebrity isn't available yet.

The key insight: the proxy controls access. Different types of proxies control different types of access:
Remote Proxy — controls access to a remote object (lives in another JVM/machine)
Virtual Proxy — controls access to an expensive resource (lazy creation)
Protection Proxy — controls access based on permissions (access rights)
The proxy implements the same interface as the real subject. The client can't tell if it's talking to the real thing or a proxy. That's the power of programming to interfaces + composition.
Implementation
Remote Proxy — Making GumballMachine a Remote Service
Using Java RMI, we turn our GumballMachine into a remote service and create a proxy that the monitor can use from anywhere on the network.
Step 1: Remote Interface
import java.rmi.*; public interface GumballMachineRemote extends Remote { public int getCount() throws RemoteException; public String getLocation() throws RemoteException; public State getState() throws RemoteException; } // State must be Serializable! public interface State extends Serializable { public void insertQuarter(); public void ejectQuarter(); public void turnCrank(); public void dispense(); }
Step 2: GumballMachine as Service
public class GumballMachine extends UnicastRemoteObject implements GumballMachineRemote { // transient = don't serialize this! transient State state; String location; int count; public GumballMachine( String location, int count) throws RemoteException { // constructor code... } // getCount(), getState(), getLocation() // all already implemented! }
The GumballMachine now extends UnicastRemoteObject and implements our remote interface. RMI handles creating the proxy (stub) and skeleton automatically. We just need to register it.
Integration
Wiring It Up — Registration & Remote Monitor
We register the gumball machine with RMI registry, then the monitor looks up the proxy and uses it just like a local object.
Server Side — Register
// Register with RMI registry try { GumballMachineRemote gm = new GumballMachine( "seattle.mightygumball.com", 250); Naming.rebind( "//seattle.mightygumball.com" + "/gumballmachine", gm); } catch (Exception e) { e.printStackTrace(); } // RMI swaps the real object for a // stub (proxy) in the registry!
Client Side — Monitor
// Monitor uses the INTERFACE public class GumballMonitor { GumballMachineRemote machine; public void report() { try { System.out.println( machine.getLocation()); System.out.println( machine.getCount()); System.out.println( machine.getState()); } catch (RemoteException e) { e.printStackTrace(); } } } // Look up the proxy: GumballMachineRemote gm = (GumballMachineRemote) Naming.lookup( "rmi://seattle.../gumballmachine");
// Output — The CEO is thrilled! Gumball Machine: seattle.mightygumball.com Current inventory: 187 gumballs Current state: waiting for quarter Gumball Machine: boulder.mightygumball.com Current inventory: 44 gumballs Current state: waiting for turn of crank
The monitor code barely changed! It uses the interface and doesn't care whether the object is local or remote. The proxy handles all the networking transparently.
Proxy Variant
Virtual Proxy — The CD Cover Viewer
A different kind of access control: the Virtual Proxy stands in for an expensive-to-create object, displaying a placeholder until the real object is ready.
Imagine a CD cover viewer app. Loading images from the network takes time. Without a proxy, the entire UI freezes while waiting. A Virtual Proxy displays "Loading..." while the real image loads in a background thread.
ImageProxy Code
class ImageProxy implements Icon { ImageIcon imageIcon; // the real subject URL imageURL; public void paintIcon(Component c, Graphics g, int x, int y) { if (imageIcon != null) { // Image loaded — delegate! imageIcon.paintIcon(c, g, x, y); } else { // Still loading — show placeholder g.drawString( "Loading CD cover...", x+300, y+190); // Start background thread... retrievalThread = new Thread(() -> { imageIcon = new ImageIcon(imageURL); c.repaint(); }); } } }

Proxy Variant
Protection Proxy — The Matchmaking Service
Yet another type of access control: a Protection Proxy controls who can call which methods, based on the caller's role.
Objectville's dating service has a PersonBean with getters and setters. Problem: users are inflating their own "Hot or Not" ratings and changing other people's profiles. We need different access rules for different callers.
Owner Proxy — Your Own Profile
public class OwnerInvocationHandler implements InvocationHandler { PersonBean person; public Object invoke(Object proxy, Method method, Object[] args) { if (method.getName() .startsWith("get")) { // Owners can get anything return method.invoke(person, args); } else if (method.getName() .equals("setHotOrNotRating")) { // Can't rate yourself! throw new IllegalAccessException(); } else if (method.getName() .startsWith("set")) { // Owners can set their own info return method.invoke(person, args); } } }
Non-Owner Proxy — Someone Else's Profile
public class NonOwnerInvocationHandler implements InvocationHandler { PersonBean person; public Object invoke(Object proxy, Method method, Object[] args) { if (method.getName() .startsWith("get")) { // Anyone can view profiles return method.invoke(person, args); } else if (method.getName() .equals("setHotOrNotRating")) { // Others CAN rate you return method.invoke(person, args); } else if (method.getName() .startsWith("set")) { // Can't change someone else's info! throw new IllegalAccessException(); } } }

Java's Dynamic Proxy creates the proxy class at runtime from interfaces you supply. You just write the InvocationHandler to control access. The proxy is generated by Proxy.newProxyInstance().
Trade-offs
Proxy vs. Decorator vs. Adapter — What's the Difference?
They all "wrap" objects. But their INTENT is completely different.
Structural Comparison
Proxy
Controls access
Same interface
May create subject
Subject may not exist yet
Decorator
Adds behavior
Same interface
Always wraps existing obj
Can stack recursively
Adapter
Changes interface
Different interface
Always wraps existing obj
Converts one API → another
Key Distinctions
// PROXY — controls access proxy.request() → "Should I allow this call?" → subject.request() // maybe // DECORATOR — adds behavior decorator.request() → doExtraStuff() // always → wrapped.request() // always → doMoreExtraStuff() // ADAPTER — changes interface adapter.newMethod() → adaptee.oldMethod() // different API
Decorator never denies access — it always delegates AND adds behavior. Proxy may deny access entirely (protection proxy), create the subject lazily (virtual proxy), or forward across a network (remote proxy). Adapter changes the interface; proxy and decorator keep it the same.
The Proxy Zoo
Proxy Variants in the Wild
The Proxy Pattern manifests in many forms. All share the same core structure: a stand-in that controls access.
🌐 Remote Proxy
Stand-in for an object in a different JVM. Handles network communication.
⏳ Virtual Proxy
Stand-in for an expensive object. Creates it lazily when needed.
🛡️ Protection Proxy
Controls access based on caller's permissions/role.
🔥 Firewall Proxy
Protects subject from "bad" clients on the network.
💾 Caching Proxy
Caches results of expensive operations. Shares results across clients.
📋 Smart Reference
Adds actions when subject is referenced (e.g., counting references).
🔄 Synchronization Proxy
Provides safe multi-threaded access to a subject.
📝 Copy-On-Write Proxy
Defers copying until modification is needed. (Virtual Proxy variant.)
All these variants share the same pattern: they implement the same interface as the subject and control access in some way. The proxy structure is the same — only the type of access control varies.
