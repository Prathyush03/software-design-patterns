Chapter 7 — Adapter & Facade Patterns
The Brief Incompatible Interfaces Everywhere

You've built a system that uses Ducks. But now Turkeys show up, and they speak a completely different language. Meanwhile, your new home theater has so many components that watching a movie feels like piloting a spaceship.

Problem 1: The Duck System
public interface Duck { public void quack(); public void fly(); } public class MallardDuck implements Duck { public void quack() { System.out.println("Quack"); } public void fly() { System.out.println("I'm flying"); } }
The Newcomer: Turkeys
public interface Turkey { public void gobble(); // NOT quack! public void fly(); // short distance only } public class WildTurkey implements Turkey { public void gobble() { System.out.println("Gobble gobble"); } public void fly() { System.out.println("I'm flying a short distance"); } }
The setup: Your client code is written against the Duck interface. Turkeys have gobble() instead of quack(), and they can only fly short distances. You can't change either interface. How do you make a Turkey work where a Duck is expected?
First Attempt Just Use the Turkey Directly?

A naive developer might think: "I'll just modify my client code to handle both Ducks and Turkeys!"

public class DuckSimulator { public static void main(String[] args) { MallardDuck duck = new MallardDuck(); WildTurkey turkey = new WildTurkey(); System.out.println("The Duck says..."); duck.quack(); // works fine duck.fly(); // works fine System.out.println("The Turkey says..."); turkey.gobble(); // different method name! turkey.fly(); // different behavior! // Can't pass turkey to testDuck()! testDuck(turkey); // COMPILE ERROR! } static void testDuck(Duck duck) { duck.quack(); // Turkey doesn't have quack()! duck.fly(); } }

Compile error! The client's testDuck() method expects a Duck. A Turkey is not a Duck. We can't change the client, and we can't change the Turkey. We're stuck... or are we?
Oh No! Diagnosing With Known Principles

Let's use the OO principles we've already learned to figure out what's going wrong and what to do.

// Our client code — hardcoded to ONE interface static void testDuck(Duck duck) { // ❌ "Program to interfaces" duck.quack(); // — but Turkey has a DIFFERENT interface! duck.fly(); } // If we try to support both directly: static void testBird(Object bird) { // ❌ "Encapsulate what varies" if (bird instanceof Duck) { // — change is SPREAD across client ((Duck)bird).quack(); } else if (bird instanceof Turkey) { ((Turkey)bird).gobble(); // ❌ "Favor composition" } // — we're not COMPOSING, we're type-checking }
Principle Violation
Program to interfaces, not implementations Client is coded to Duck — but now there's a second valid interface (Turkey). We need to bridge them.
Encapsulate what varies The varying part is the interface translation. Currently it's scattered as instanceof checks in the client.
Favor composition over inheritance We can't make Turkey extend Duck (wrong hierarchy). We need to WRAP the Turkey and delegate.
Three principles violated. Each one tells us exactly what to fix. Let's apply them one at a time.
Design Principle Encapsulate What Varies
OO Principle
"Identify the aspects of your application that vary and separate them from what stays the same."

What varies? The interface translation from Turkey methods to Duck methods. Let's extract that into its own class.

Before — Translation in client
static void testBird(Object bird) { if (bird instanceof Duck) { ((Duck)bird).quack(); } else if (bird instanceof Turkey){ ((Turkey)bird).gobble(); } } // Translation logic INSIDE client // Every new bird = more instanceof!
After — Separate adapter class
public class TurkeyAdapter { Turkey turkey; public TurkeyAdapter(Turkey t) { this.turkey = t; } public void quack() { turkey.gobble(); // translate! } } // Translation is ENCAPSULATED

Design Principle Program to Interfaces
OO Principle
"Program to an interface, not an implementation."

The adapter must implement the TARGET interface — Duck — so the client can use it without knowing it's a Turkey underneath.

Before — No interface
public class TurkeyAdapter { Turkey turkey; // Has quack() but DOESN'T // implement Duck! // testDuck(Duck) won't // accept this! public void quack() { turkey.gobble(); } }

public class TurkeyAdapter implements Duck { Turkey turkey; // NOW the client can use // this wherever Duck is // expected! public void quack() { turkey.gobble(); } public void fly() { for(int i=0; i < 5; i++) turkey.fly(); } }

Now TurkeyAdapter IS-A Duck! The client can call testDuck(turkeyAdapter) without any changes. But how does the adapter talk to the Turkey? Through composition — which is exactly the next principle.
Design Principle Favor Composition Over Inheritance
OO Principle
"Favor composition over inheritance."

The adapter WRAPS the Turkey (HAS-A), it doesn't extend it. This means any Turkey subclass works with our adapter.

Inheritance approach (bad)
// Can't do this in Java anyway // (no multiple inheritance) public class TurkeyAdapter extends WildTurkey extends MallardDuck { // LOCKED to WildTurkey only // Can't adapt DomesticTurkey // or any other Turkey! }
Composition approach (good)
public class TurkeyAdapter implements Duck { Turkey turkey; // HAS-A Turkey public TurkeyAdapter(Turkey t) { this.turkey = t; } // Works with ANY Turkey! // WildTurkey, DomesticTurkey, // even turkeys we haven't // written yet! }

Revelation The Adapter Pattern!

By applying three principles we already knew, we independently arrived at the Adapter Pattern. It's like a real-world power adapter — converting one interface into another!

Pattern Definition
The Adapter Pattern converts the interface of a class into another interface the clients expect. Adapter lets classes work together that couldn't otherwise because of incompatible interfaces.

The analogy: Think of a European power adapter. The wall outlet (Adaptee) has one interface. Your laptop (Client) expects another. The adapter sits in between, converting one into the other. The client never knows the difference.
Implementation Test Drive — The Complete Adapter

Let's wire everything together and see the adapter in action.

public class DuckTestDrive { public static void main(String[] args) { MallardDuck duck = new MallardDuck(); WildTurkey turkey = new WildTurkey(); Duck turkeyAdapter = new TurkeyAdapter(turkey); System.out.println("The Turkey says..."); turkey.gobble(); turkey.fly(); System.out.println("The Duck says..."); testDuck(duck); System.out.println("The TurkeyAdapter says..."); testDuck(turkeyAdapter); // Turkey in Duck's clothing! } static void testDuck(Duck duck) { duck.quack(); // calls gobble() behind the scenes duck.fly(); // calls turkey.fly() × 5 } }

It works! The testDuck() method never knows it's talking to a Turkey. The adapter handles all the translation. Now let's look at a real-world Java example...
Real World Adapting Enumeration to Iterator

Java's old collections used Enumeration. New ones use Iterator. We need legacy code to work with new code. Sound familiar?

Old World: Enumeration
«interface» Enumeration
hasMoreElements()
nextElement()
New World: Iterator
«interface» Iterator
hasNext()
next()

public class EnumerationIterator implements Iterator { Enumeration enumeration; // HAS-A adaptee public EnumerationIterator(Enumeration enumeration) { this.enumeration = enumeration; } public boolean hasNext() { return enumeration.hasMoreElements(); // direct map } public Object next() { return enumeration.nextElement(); // direct map } public void remove() { throw new UnsupportedOperationException(); // Enumeration is read-only — best we can do! } }
Imperfect adapter: The remove() method has no equivalent in Enumeration. We throw UnsupportedOperationException — an honest admission. Not every translation is perfect, but the adapter still provides the bridge.

Diagnose What Principles Are Violated?

Let's diagnose this complex subsystem using our principles.

// Client code for watching a movie: popper.on(); popper.pop(); lights.dim(10); screen.down(); projector.on(); projector.setInput(dvd); // ❌ Client knows about 6 classes projector.wideScreenMode(); // ❌ Client knows internal details amp.on(); // ❌ Client is tightly coupled to amp.setDvd(dvd); // every single subsystem class amp.setSurroundSound(); amp.setVolume(5); dvd.on(); dvd.play(movie);
PrincipleViolation
Encapsulate what varies The "watch movie" workflow is scattered across the client. If the projector brand changes, the CLIENT must change.
Strive for loosely coupled designs The client is coupled to 6 different subsystem classes! Any change to any of them breaks the client.
Tight coupling at scale. Let's apply the principles to fix this — encapsulate the complexity and reduce coupling.

The Facade Pattern: provides a unified, simplified interface to a set of interfaces in a subsystem. It doesn't "hide" the subsystem — you can still access those classes directly if you need low-level control.
Implementation Building the HomeTheaterFacade

The Facade uses composition to hold references to all subsystem components, then delegates to them in the right order.

public class HomeTheaterFacade { Amplifier amp; Tuner tuner; DvdPlayer dvd; CdPlayer cd; Projector projector; TheaterLights lights; Screen screen; PopcornPopper popper; public HomeTheaterFacade(/* all components */) { // assign each to instance variable } public void watchMovie(String movie) { System.out.println("Get ready to watch a movie..."); popper.on(); popper.pop(); lights.dim(10); screen.down(); projector.on(); projector.wideScreenMode(); amp.on(); amp.setDvd(dvd); amp.setSurroundSound(); amp.setVolume(5); dvd.on(); dvd.play(movie); } public void endMovie() { popper.off(); lights.on(); screen.up(); projector.off(); amp.off(); dvd.stop(); dvd.eject(); dvd.off(); } }
Key insight: The Facade doesn't add new functionality — it orchestrates existing subsystem functionality into convenient high-level operations. The subsystem classes don't even know the Facade exists.
New Principle The Principle of Least Knowledge

The Facade Pattern naturally leads us to a new design principle that emerges from what we just built.

New OO Principle
"Talk only to your immediate friends." (Also known as the Law of Demeter)

The principle says: from any method, only call methods on (1) the object itself, (2) objects passed as parameters, (3) objects you create, and (4) your component objects.

Violates the Principle
public float getTemp() { return station .getThermometer() .getTemperature(); } // Calling a method on an object // returned from ANOTHER call! // You now depend on TWO classes.
Follows the Principle
public float getTemp() { return station .getTemperature(); } // Ask the station directly. // Station delegates internally. // You only depend on ONE class.
This principle is a consequence of the Facade Pattern. The Facade gives the client ONE friend instead of six. The client talks only to the Facade, and the Facade talks to the subsystem. Fewer dependencies = less fragile system.
Trade-offs Adapter vs. Facade vs. Decorator

All three patterns wrap objects. But their intents are completely different.

Facade Simplify a complex interface Yes — makes it simpler Many (entire subsystem)
Decorator Add new responsibilities No — keeps same interface One

Key distinction: the difference isn't how many classes they wrap — it's their intent. An Adapter wraps to convert. A Facade wraps to simplify. A Decorator wraps to add behavior. Choose based on what you're trying to achieve.
Trade-offs The Cost of Least Knowledge

The Principle of Least Knowledge reduces coupling, but it's not free.

Advantages
Reduced dependencies Fewer classes coupled together. Changes don't cascade. Simpler maintenance Studies show this principle reduces software maintenance costs. More flexible Can swap subsystem components without touching client code.
Disadvantages
More wrapper classes More "middle-man" code to write and maintain. Complexity & performance Extra layers of indirection. More method calls to traverse. Can feel like overkill System.out.println() violates the principle — should we care? // Sometimes pragmatism wins.
All principles involve trade-offs. The Principle of Least Knowledge is a guideline, not a law. Apply it where coupling is causing real pain, not dogmatically everywhere.
