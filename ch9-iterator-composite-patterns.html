Ch. 9 — Iterator & Composite

Phase 1 · The Brief
Two Menus, One Waitress
Objectville Diner and Pancake House have merged. Both restaurants have their own menu implementation — and neither chef will budge.
Lou (Pancake House) stores menu items in an ArrayList. Mel (Diner) uses a plain Array. They agreed on a common MenuItem class, but that's where cooperation ends. Our job: build a Waitress that can print both menus.

The goal: build a Waitress class that can iterate over both menus to print items, check for vegetarian options, etc. — without caring how each menu stores its items.
Phase 1 · First Attempt
The Brute-Force Waitress
We call getMenuItems() on both menus and write loops. Simple, right?
public class Waitress { PancakeHouseMenu pancakeHouseMenu; DinerMenu dinerMenu; public void printMenu() { ArrayList breakfastItems = pancakeHouseMenu.getMenuItems(); MenuItem[] lunchItems = dinerMenu.getMenuItems(); for (int i = 0; i < breakfastItems.size(); i++) { MenuItem m = (MenuItem) breakfastItems.get(i); System.out.println(m.getName() + " " + m.getPrice()); } for (int i = 0; i < lunchItems.length; i++) { MenuItem m = lunchItems[i]; System.out.println(m.getName() + " " + m.getPrice()); } } }
It compiles. It runs. The developer feels confident. But then a Café wants to join the merger — they use a Hashtable. Now we need three loops, three concrete types, and the Waitress keeps growing...
Feels fine now — but what happens when a 4th restaurant joins? We'd need a 4th loop, a 4th concrete reference. Every menu change means changing the Waitress.
Phase 1 · Oh No!
Diagnosis: Which Principles Are Violated?
Let's use the OO principles we already know as a diagnostic checklist against our naive Waitress code.
public void printMenu() { ArrayList breakfastItems = pancakeHouseMenu.getMenuItems(); // ❌ Encapsulation: internal collection exposed MenuItem[] lunchItems = dinerMenu.getMenuItems(); // ❌ Encapsulation: internal collection exposed for (int i = 0; i < breakfastItems.size(); i++) // ❌ Coding to concrete ArrayList ... for (int i = 0; i < lunchItems.length; i++) // ❌ Coding to concrete Array ... // ❌ Open-Closed: adding a 3rd menu = modifying Waitress // ❌ No common interface for menus }
Violations Found
1. Encapsulate what varies Iteration logic varies per collection type — but it's baked into Waitress. 2. Program to interfaces Waitress references ArrayList and MenuItem[] directly — concrete types. 3. Open-Closed Principle Every new menu means modifying Waitress code. 4. Single Responsibility Waitress manages iteration AND uses menu data — two reasons to change.

Phase 2 · Apply Principle
Principle 1: Encapsulate What Varies
What varies between the two menus? The iteration logic. Let's extract it into its own object.
Design Principle
"Encapsulate what varies — identify the aspects that vary and separate them from what stays the same."
Before — iteration mixed in
// In Waitress — two different loops! for (int i = 0; i < breakfastItems.size(); i++) MenuItem m = (MenuItem) breakfastItems.get(i); for (int i = 0; i < lunchItems.length; i++) MenuItem m = lunchItems[i];
After — iteration encapsulated
// New idea: an Iterator object! Iterator iter = menu.createIterator(); while (iter.hasNext()) { MenuItem m = (MenuItem) iter.next(); } // ONE loop style for ALL menus!

Phase 2 · Apply Principle
Principle 2: Program to Interfaces
Define an Iterator interface so every menu can provide its own iterator implementation.
Design Principle
"Program to an interface, not an implementation."
Before — concrete types
// Waitress bound to concrete types ArrayList breakfastItems = ... MenuItem[] lunchItems = ...
After — interface
public interface Iterator { boolean hasNext(); Object next(); } // Now Waitress only knows Iterator! Iterator iter = menu.createIterator();
Concrete: DinerMenuIterator
public class DinerMenuIterator implements Iterator { MenuItem[] items; int position = 0; public Object next() { MenuItem m = items[position]; position++; return m; } public boolean hasNext() { return position < items.length && items[position] != null; } }

Phase 2 · Apply Principle
Principle 3: Open for Extension, Closed for Modification
Create a Menu interface so the Waitress can handle any number of menus without code changes.
Design Principle
"Classes should be open for extension but closed for modification."
Before — concrete menu refs
public class Waitress { PancakeHouseMenu pancakeHouseMenu; DinerMenu dinerMenu; // Adding CafeMenu? Change this class! }
After — Menu interface
public interface Menu { Iterator createIterator(); } public class Waitress { ArrayList<Menu> menus; // Add new menus? No changes needed! }
Updated Waitress
public class Waitress { ArrayList<Menu> menus; public void printMenu() { Iterator menuIter = menus.iterator(); while (menuIter.hasNext()) { Menu menu = (Menu) menuIter.next(); printMenu(menu.createIterator()); } } void printMenu(Iterator iter) { while (iter.hasNext()) { MenuItem m = (MenuItem) iter.next(); System.out.println(m.getName()); } } }

Phase 3 · The Revelation
Wait... This Has a Name!
By applying known principles one at a time, we've independently arrived at the Iterator Pattern.
Iterator Pattern — Definition
"The Iterator Pattern provides a way to access the elements of an aggregate object sequentially without exposing its underlying representation."
Think of it like a waiter in a real restaurant: you don't need to go into the kitchen and look through the fridge yourself. The waiter brings items to you one at a time. You don't care if they're stored on shelves, in drawers, or stacked in a pile — you just say "next, please."

New Principle Unlocked!
Single Responsibility Principle
"A class should have only one reason to change."
By pulling iteration out of the aggregate, we gave each class exactly one job. The aggregate manages the collection; the iterator manages traversal. This is the Single Responsibility Principle — and it emerged from applying the pattern, not the other way around.
The aha moment: Encapsulate what varies → Program to interfaces → Open-Closed → and you get the Iterator Pattern + the Single Responsibility Principle as a bonus.

Phase 3b · The Second Revelation
The Composite Pattern
Compose objects into tree structures to represent part-whole hierarchies. Treat individual objects and compositions uniformly.
Composite Pattern — Definition
"The Composite Pattern allows you to compose objects into tree structures to represent part-whole hierarchies. Composite lets clients treat individual objects and compositions of objects uniformly."
The key insight: create a MenuComponent abstract class that both MenuItem (leaf) and Menu (composite) extend. The Waitress calls print() on the top-level component and the recursion handles the rest.

The Magic: Recursive print()
// Menu's print() — the recursive bit public void print() { System.out.println(getName()); System.out.println("----------"); Iterator iter = menuComponents.iterator(); while (iter.hasNext()) { MenuComponent mc = (MenuComponent) iter.next(); mc.print(); // could be a Menu! } } // Waitress — incredibly simple public void printMenu() { allMenus.print(); // that's it! }
The beauty: If mc is a MenuItem, it prints itself. If it's a Menu, it prints its header and then recursively prints all its children. The Waitress doesn't need to know the difference — that's transparency.
Phase 4 · Implementation
Building the Composite Menu
The MenuComponent base class, MenuItem leaf, and Menu composite — all wired up.
MenuComponent (abstract)
public abstract class MenuComponent { // "composite" methods — default: throw public void add(MenuComponent mc) { throw new UnsupportedOperationException(); } public void remove(MenuComponent mc) { throw new UnsupportedOperationException(); } public MenuComponent getChild(int i) { throw new UnsupportedOperationException(); } // "leaf" methods — default: throw public String getName() { throw new UnsupportedOperationException(); } public double getPrice() { throw new UnsupportedOperationException(); } public void print() { throw new UnsupportedOperationException(); } }
MenuItem (leaf)
public class MenuItem extends MenuComponent { String name; String description; boolean vegetarian; double price; // constructor, getters... public void print() { System.out.print(" " + getName()); if (isVegetarian()) System.out.print("(v)"); System.out.println(", " + getPrice()); } }
Menu (composite)
public class Menu extends MenuComponent { ArrayList menuComponents = new ArrayList(); String name, description; public void add(MenuComponent mc) { menuComponents.add(mc); } public void print() { System.out.println(getName()); Iterator i = menuComponents.iterator(); while (i.hasNext()) { MenuComponent mc = (MenuComponent) i.next(); mc.print(); } } }
Notice: Iterator and Composite work together beautifully. The Menu uses an Iterator internally to traverse its children, and each child can be either a leaf or another composite. The two patterns complement each other.
Phase 5 · Trade-offs
Transparency vs. Safety
The Composite Pattern puts BOTH leaf and composite methods in one interface. Isn't that a violation of Single Responsibility?
Sharp eye! The MenuComponent has both leaf operations (getPrice(), isVegetarian()) and composite operations (add(), remove(), getChild()). This is a deliberate trade-off.
Option A: Transparency
✓ Client treats everything uniformly ✓ No instanceof checks needed ✓ Simple client code ✗ Leaf has methods that don't apply ✗ Calling add() on a MenuItem? Exception! ✗ Technically violates Single Responsibility This is the approach the book takes.
Option B: Safety
✓ No meaningless methods on leaf ✓ Compile-time type safety ✓ Clean Single Responsibility ✗ Client must check types before calling ✗ Needs instanceof and casting ✗ Loses uniform treatment Separate interfaces for Leaf & Composite.
The lesson: Principles are guides, not laws. Sometimes we intentionally trade one principle (Single Responsibility) for a practical benefit (transparency). The key is making the trade-off consciously, not accidentally.
Phase 4 · Advanced
The CompositeIterator: External Traversal
What if the Waitress needs to filter items (e.g., vegetarian only) across ALL menus? She needs an external iterator over the entire tree.
The internal print() recursion works great for printing everything. But for selective operations — like "show me only vegetarian items across all sub-menus" — we need an external CompositeIterator that walks the entire tree, returning one leaf at a time.
CompositeIterator (uses a Stack)
public class CompositeIterator implements Iterator { Stack stack = new Stack(); public CompositeIterator(Iterator iter) { stack.push(iter); } public boolean hasNext() { if (stack.empty()) return false; Iterator iter = (Iterator) stack.peek(); if (!iter.hasNext()) { stack.pop(); return hasNext(); // recursive! } return true; } public Object next() { Iterator iter = (Iterator) stack.peek(); MenuComponent mc = (MenuComponent) iter.next(); if (mc instanceof Menu) { stack.push(mc.createIterator()); } return mc; } }
Waitress: Vegetarian Filter
public void printVegetarianMenu() { Iterator iter = allMenus.createIterator(); while (iter.hasNext()) { MenuComponent mc = (MenuComponent) iter.next(); try { if (mc.isVegetarian()) { mc.print(); } } catch (UnsupportedOperationException) {} } }
NullIterator (for leaves)
// MenuItem has no children to iterate public class NullIterator implements Iterator { public boolean hasNext() { return false; // always done } public Object next() { return null; } }
Internal vs. External iteration: The recursive print() is internal iteration — the composite controls traversal. The CompositeIterator is external — the client (Waitress) controls traversal. Both have their uses; external gives the client more flexibility.
