Phase 1 · Oh No!

Diagnosis: what principles are we violating?

Let's pull out our OO design principle checklist and examine this code. We've learned several principles across chapters 1–7. Let's see which ones we're breaking.

Diagnostic Checklist
// ❌ PRINCIPLE: "Encapsulate what varies"
Coffee.prepareRecipe()  — controls the algorithm
Tea.prepareRecipe()     — controls the SAME algorithm
//    The algorithm structure is the SAME but it's scattered
//    across two classes. If the recipe changes, we edit BOTH.

// ❌ PRINCIPLE: "Don't duplicate code" (DRY)
Coffee.boilWater()  == Tea.boilWater()     // IDENTICAL
Coffee.pourInCup()  == Tea.pourInCup()     // IDENTICAL
//    Two copies of the same code. Bug fix = fix twice.

// ❌ PRINCIPLE: "Program to interfaces/abstractions"
// No shared type! A Waitress can't treat Coffee and Tea
// the same way. No polymorphism possible.

// ❌ PRINCIPLE: "Closed for modification"
// Adding a new beverage (HotChocolate) means writing
// the ENTIRE algorithm again from scratch.

Phase 2 · Apply Principle 1

Pull up common code into a superclass

Design Principle
"Encapsulate what varies" + DRY — extract common code into one place

Phase 2 · Apply Principle 2

Generalize the algorithm steps

Design Principle
"Program to abstractions" — find the common structure even when details differ

Look at the two recipes side by side. Steps 1 and 3 are identical. Steps 2 and 4 are different in detail but same in concept: "brew" the beverage and "add condiments." If we generalize the names, the algorithm is identical!

Before — Different method names
// Coffee
void prepareRecipe() {
  boilWater();
  brewCoffeeGrinds();
  pourInCup();
  addSugarAndMilk();
}

// Tea
void prepareRecipe() {
  boilWater();
  steepTeaBag();
  pourInCup();
  addLemon();
}
After — Generalized names
// Both Coffee AND Tea:
void prepareRecipe() {
  boilWater();
  brew();          // generalized!
  pourInCup();
  addCondiments(); // generalized!
}

// Now the algorithm is IDENTICAL
// for both beverages.
// We can pull prepareRecipe()
// into the superclass!
By renaming brewCoffeeGrinds() → brew() and steepTeaBag() → brew(), and addSugarAndMilk()/addLemon() → addCondiments(), the algorithm structure becomes one thing. But who implements brew() and addCondiments()? The subclasses — via abstract methods!
Phase 2 · Apply Principle 3

Lock down the algorithm with final

Design Principle
"Closed for modification" — the algorithm structure must not change; only the steps can vary

We move prepareRecipe() into the superclass and declare it final. No subclass can override it. The steps that vary — brew() and addCondiments() — are declared abstract.

Phase 3 · Revelation

Wait… what we just built has a name!

By applying our principles step by step, we've independently arrived at the Template Method Pattern. The prepareRecipe() method is the template method — it defines the skeleton of an algorithm, deferring some steps to subclasses.

The Template Method Pattern
Define the skeleton of an algorithm in a method, deferring some steps to subclasses. Template Method lets subclasses redefine certain steps without changing the algorithm's structure.
The Template Method — Annotated
public abstract class CaffeineBeverage {

  final void prepareRecipe() {  // THE template method
    boilWater();          // concrete step (shared)
    brew();               // abstract step (subclass)
    pourInCup();          // concrete step (shared)
    addCondiments();      // abstract step (subclass)
  }

  abstract void brew();            // subclass MUST provide
  abstract void addCondiments();   // subclass MUST provide

  void boilWater() { ... }         // shared — in superclass
  void pourInCup() { ... }         // shared — in superclass
}
Think of it like a recipe card with blanks: "1. Boil water, 2. _______, 3. Pour, 4. _______." The card controls the process; you just fill in the blanks.
Phase 3 · New Principle

The Hollywood Principle: "Don't call us, we'll call you"

Notice something about our design? The superclass calls the subclass methods — never the other way around. CaffeineBeverage says to Tea and Coffee: "Don't call me, I'll call you when I need your brew() step."

New Design Principle
The Hollywood Principle — Don't call us, we'll call you.
How it works
CaffeineBeverage
High-level component
final prepareRecipe()
→ calls brew()
→ calls addCondiments()
△ "I'll call you when I need you"
Tea
Low-level component
brew()
addCondiments()
Coffee
Low-level component
brew()
addCondiments()
Tea & Coffee never call CaffeineBeverage directly
The Hollywood Principle prevents "dependency rot" — where high-level and low-level components become tangled in circular dependencies. It's related to the Dependency Inversion Principle (ch. 4), but is specifically a technique for frameworks: let low-level components hook in, but keep control at the top.
Phase 4 · Hooks

Hooks: optional steps in the algorithm

What if a step is optional? We don't want to force subclasses to implement it. That's where hooks come in — methods with a default (often empty) implementation that subclasses may override.

Hook in the template
abstract class CaffeineBeverage {

  final void prepareRecipe() {
    boilWater();
    brew();
    pourInCup();
    if (customerWantsCondiments()) {
      addCondiments();
    }
  }

  abstract void brew();
  abstract void addCondiments();

  boolean customerWantsCondiments() {
    return true; // hook! default=yes
  }
}
Subclass overrides the hook
public class Coffee
    extends CaffeineBeverage {

  public void brew() {
    System.out.println(
      "Dripping through filter");
  }
  public void addCondiments() {
    System.out.println(
      "Adding Sugar and Milk");
  }

  public boolean
      customerWantsCondiments() {
    String answer = getUserInput();
    return answer.startsWith("y");
  }
}
Rule of thumb: Use abstract methods when the subclass must provide an implementation. Use hooks when it's optional. Hooks keep the subclass's burden light while still allowing control over the algorithm flow.
Phase 4 · The Formal Pattern

Template Method: the formal class diagram

Here's the general structure that applies to any template method — not just beverages.

General Template Method Structure
«abstract» AbstractClass
final templateMethod()
→ primitiveOp1()
→ primitiveOp2()
→ concreteOp()
→ hook()
abstract primitiveOp1()
abstract primitiveOp2()
concreteOperation()

ConcreteClass
primitiveOp1() ← required
primitiveOp2() ← required

Three kinds of methods live in the abstract class: abstract (subclass must implement), concrete (shared, optionally final), and hooks (empty default, subclass may override).

Phase 5 · In the Wild

Template Method is everywhere

This pattern appears throughout the Java API. Here's one you use all the time without realizing it: Arrays.sort().

java.util.Arrays (the template)
// Arrays.sort() is the template method
// It controls the sort algorithm (mergeSort)
// but delegates COMPARISON to YOUR class

public static void sort(Object[] a) {
  // ... mergeSort algorithm ...
  for (...) {
    if (a[i].compareTo(a[j]) > 0)
      swap(a, i, j);
  }
}

// compareTo() is the "primitive operation"
// — YOU fill in this step!
Duck.java (you implement compareTo)
public class Duck
    implements Comparable<Duck> {

  String name;
  int weight;

  public int compareTo(Duck other) {
    if (this.weight < other.weight)
      return -1;
    else if (this.weight == other.weight)
      return 0;
    else return 1;
  }
}

// Arrays.sort(ducks) — now works!
Note the twist: Arrays.sort() uses composition (Comparable) rather than inheritance. Same idea — the framework controls the algorithm, you supply the varying step — but with an interface instead of a superclass. Other examples: JFrame.paint() (hook), Applet.init() (hook), InputStream.read().
Phase 5 · Trade-offs

Template Method vs. Strategy: the showdown

Both patterns encapsulate algorithms. But they take fundamentally different approaches. Here's how they compare.

Template Method (inheritance)Strategy (composition)
Algorithm defined in superclass Algorithm defined in separate strategy objects
Subclasses fill in specific steps Client selects the entire algorithm
More code reuse — shared steps stay in superclass More flexible — swap algorithms at runtime
Less flexible — locked at compile time Less code reuse — each strategy has full algorithm
Subclass depends on superclass Strategy is independent — no dependency
When to choose: Template Method when the algorithm structure is fixed and you only need to vary a few steps. Strategy when you need completely interchangeable algorithms or runtime swapping. Factory Method is actually a specialization of Template Method!
